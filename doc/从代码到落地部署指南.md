# 地推渠道活码系统端到端部署指南（从代码到落地）

## 1. 文档定位

这份文档是本仓库的部署主入口，主要说明代码如何构建、部署、验收和回滚。研发可以按它自己部署，交给运维时也可以直接作为执行依据。

适用范围：

1. 单机 Ubuntu 私有云服务器：`Docker Compose + MySQL + MinIO/OBS`
2. 公司 Kubernetes 集群：`镜像构建 + Registry + ConfigMap/Secret + Ingress`

关联文档：

1. `doc/K8S-OBS部署清单.md`：K8S 摘要清单
2. `doc/Railway部署说明.md`：Railway 历史测试环境说明
3. `doc/local-dev.md`：本地开发启动方式

## 2. 项目部署全景

### 2.1 代码结构与运行产物

| 目录 | 作用 | 最终产物 |
| --- | --- | --- |
| `backend/` | Spring Boot 后端服务 | Docker 镜像 / JAR |
| `frontend/` | Vue 3 + Vite 前端 | Docker 镜像 / 静态文件 |
| `sql/` | 初始化与升级 SQL | MySQL 数据库脚本 |
| `deploy/k8s/base/` | K8S 基础 YAML | Namespace / Deployment / Service / Ingress 等 |
| `doc/` | 技术与部署文档 | 研发和运维执行依据 |

### 2.2 运行时拓扑

线上链路大致如下：

1. 用户访问前端域名
2. 前端静态资源由 Nginx 提供
3. 浏览器请求 `/api` 时转发到 Spring Boot 后端
4. 后端连接 MySQL 存储业务数据
5. 后端连接 MinIO 或 OBS（S3 兼容方式）存储二维码图片
6. 微信 JS-SDK 相关能力依赖公众号配置和出口网络

### 2.3 外部依赖矩阵

| 依赖项 | 是否必需 | 说明 |
| --- | --- | --- |
| Ubuntu 22.04+ | 是 | 单机部署主机操作系统 |
| Docker Engine + Compose Plugin | 单机必需 | 单机部署执行环境 |
| Kubernetes 集群 | K8S 必需 | 公司集群部署环境 |
| MySQL 5.7+ / 8.0+ | 是 | 业务数据存储 |
| MinIO 或 OBS/S3 兼容存储 | 是 | 二维码图片存储 |
| Gitee 仓库 | 是 | 代码托管与 CI/CD 源头 |
| 镜像仓库 | K8S 推荐 | 存储前后端镜像 |
| 域名 / HTTPS 证书 | 生产推荐 | 统一公网入口 |
| 微信公众号配置 | 视业务需要 | JS-SDK 和跳转相关能力 |

## 3. 部署前必须确认的决策

开始之前，先把下面几个关键决定确认下来。很多部署问题不是出在命令本身，而是环境边界没有提前说清楚。

### 3.1 选单机还是选 K8S

| 方案 | 适用场景 | 优点 | 代价 |
| --- | --- | --- | --- |
| 单机 Ubuntu | 测试环境、轻量生产、先跑通 | 成本低、步骤少、易排查 | 扩展性一般、容灾弱 |
| K8S 集群 | 公司正式生产、要纳入统一运维 | 标准化、易扩容、利于 CI/CD | 前置条件多、配置更复杂 |

建议做法：

1. 研发自测、业务验证，先走单机 Ubuntu
2. 公司正式生产，走 K8S

### 3.2 对象存储选 MinIO 还是 OBS

| 方案 | 适用场景 | 当前仓库支持方式 |
| --- | --- | --- |
| MinIO | 单机部署、测试环境、自建对象存储 | 直接支持 |
| 华为云 OBS / 其他 S3 兼容存储 | 公司已有对象存储 | 通过 `MINIO_*` 兼容变量接入 |

这里有一个容易误解的点：

1. 当前代码没有单独的 `obs` 实现类
2. 后端统一按 `storage.type=minio` 运行
3. 接 OBS 时，不切换实现类型，只替换 `MINIO_ENDPOINT / MINIO_ACCESS_KEY / MINIO_SECRET_KEY / MINIO_BUCKET`

### 3.3 研发自部署还是交付运维

| 执行主体 | 需要准备的内容 |
| --- | --- |
| 研发自部署 | 代码、配置、镜像、部署命令、验收结果 |
| 交付运维 | 镜像地址、Tag、环境变量、Secret、SQL、发布/回滚命令、验收标准 |

如果交付运维，至少要交付：

1. 前后端镜像地址与 Tag
2. 数据库初始化脚本和升级脚本
3. 环境变量与 Secret 清单
4. 部署命令和回滚命令
5. 验收步骤和预期结果

## 4. 从代码到制品

### 4.1 代码冻结与版本约定

发布时建议统一这几个约定：

1. `main`：生产基线
2. `develop`：集成与联调
3. 发布前在 `main` 上打 Tag，例如 `v1.0.0`
4. 镜像 Tag 使用 `v1.0.0` 或 `git short sha`

### 4.2 本地构建校验

前端校验：

```bash
cd frontend
npm ci
npm run build
```

后端目前没有 `mvnw`，但 Dockerfile 已经封装了 Maven 构建流程。生产环境直接走 Docker 即可，不需要在服务器上单独安装 Maven。

### 4.3 镜像构建

在项目根目录执行：

```bash
export TAG=v1.0.0
export REGISTRY=registry.example.com

docker build -t ${REGISTRY}/qrcode-channel/backend:${TAG} backend
docker build --build-arg VITE_API_BASE_URL=/api -t ${REGISTRY}/qrcode-channel/frontend:${TAG} frontend
```

推送镜像：

```bash
docker push ${REGISTRY}/qrcode-channel/backend:${TAG}
docker push ${REGISTRY}/qrcode-channel/frontend:${TAG}
```

### 4.4 Gitee 接入 CI/CD 时至少要准备的变量

| 变量名 | 作用 |
| --- | --- |
| `REGISTRY_SERVER` | 镜像仓库地址 |
| `REGISTRY_USERNAME` | 镜像仓库账号 |
| `REGISTRY_PASSWORD` | 镜像仓库密码 |
| `KUBE_CONFIG` | K8S 集群访问凭据 |
| `K8S_NAMESPACE` | 默认命名空间，推荐 `qrcode` |
| `DB_PASSWORD` | 数据库密码 |
| `MINIO_ACCESS_KEY` | 对象存储访问密钥 |
| `MINIO_SECRET_KEY` | 对象存储访问密钥 |
| `WECHAT_MP_SECRET` | 公众号密钥 |

无论 Gitee 流水线具体怎么配置，核心动作都差不多：

1. 拉代码
2. 构建前端
3. 构建后端镜像
4. 推送镜像
5. 更新 K8S Deployment 镜像标签
6. 等待滚动发布完成

## 5. 单机 Ubuntu 部署（主线方案）

这一节对应“一台 Ubuntu 服务器直接跑完整套服务”。适合先把链路跑通，也适合测试环境和较轻的生产场景。

### 5.1 部署拓扑

单机部署的结构很简单：

1. `frontend`：Nginx，公网入口 `80`
2. `backend`：Spring Boot，仅容器内网访问
3. `mysql`：业务数据库，仅容器内网访问
4. `minio`：对象存储，仅容器内网访问
5. `minio console`：仅宿主机本地访问 `127.0.0.1:9001`

默认的安全边界如下：

1. 只对外暴露 `80`
2. `8080 / 3306 / 9000` 不暴露公网
3. `9001` 仅绑定 `127.0.0.1`

### 5.2 服务器前置准备

建议目录：

```bash
sudo mkdir -p /opt/qrcode-channel
sudo chown -R $USER:$USER /opt/qrcode-channel
cd /opt/qrcode-channel
```

安装 Docker：

```bash
sudo apt update
sudo apt install -y ca-certificates curl git
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
newgrp docker
docker --version
docker compose version
```

防火墙建议：

```bash
sudo ufw allow 80/tcp
sudo ufw enable
sudo ufw status
```

如果单机环境需要 HTTPS，建议在宿主机前面再接一层反向代理或负载均衡。仓库里默认只处理 `80` 端口。

### 5.3 拉代码与准备配置

```bash
git clone <你的 Gitee 仓库地址>
cd qrcode-channel-test
cp .env.example .env
```

`.env` 里最关键的是下面这些变量：

| 变量 | 作用 | 单机默认建议 |
| --- | --- | --- |
| `APP_BASE_URL` | 生成二维码跳转地址的根域名 | `http://你的服务器IP或域名` |
| `APP_CORS_ALLOWED_ORIGINS` | 允许访问后端的前端域名 | 与前端入口一致 |
| `VITE_API_BASE_URL` | 前端构建时 API 基址 | `/api` |
| `DB_PASSWORD` | MySQL root 密码 | 强密码 |
| `MINIO_ROOT_USER` | 本地 MinIO 管理账号 | `minioadmin` 或自定义 |
| `MINIO_ROOT_PASSWORD` | 本地 MinIO 管理密码 | 强密码 |
| `MINIO_ENDPOINT` | 后端访问对象存储地址 | 单机默认 `http://minio:9000` |
| `MINIO_ACCESS_KEY` | 后端对象存储访问账号 | 单机默认与 `MINIO_ROOT_USER` 一致 |
| `MINIO_SECRET_KEY` | 后端对象存储访问密码 | 单机默认与 `MINIO_ROOT_PASSWORD` 一致 |
| `MINIO_BUCKET` | 二维码图片 bucket | `qrcode-images` |
| `WECHAT_MP_APPID` | 公众号 AppID | 按实际填写 |
| `WECHAT_MP_SECRET` | 公众号 Secret | 按实际填写 |

### 5.4 单机部署标准步骤

1. 校验 Compose 配置

```bash
docker compose -f docker-compose.prod.yml config
```

2. 执行部署脚本

```bash
bash deploy.sh
```

3. 查看容器状态

```bash
docker compose -f docker-compose.prod.yml ps
```

4. 查看实时日志

```bash
docker compose -f docker-compose.prod.yml logs -f
```

### 5.5 单机部署脚本做了什么

`deploy.sh` 现在负责这些动作：

1. 检查 Docker 与 `docker compose`
2. 检查 `.env` 是否存在
3. 校验 `docker-compose.prod.yml`
4. 停止旧容器
5. 重新构建并启动服务
6. 轮询 `http://127.0.0.1/api/health`
7. 健康检查失败时输出最近日志

### 5.6 MinIO 与 OBS 两种用法

#### 方案 A：单机默认 MinIO

默认情况下：

1. Compose 会启动 `minio`
2. `minio-init` 会自动创建 `qrcode-images`
3. `minio-init` 会把该 bucket 设为匿名可读
4. 后端通过 `MINIO_ENDPOINT=http://minio:9000` 访问

这种方式适合：

1. 测试环境
2. 私有云内快速落地
3. 不依赖公司对象存储时的独立部署

#### 方案 B：外部 OBS / S3 兼容存储

如果公司已经有 OBS 或其他 S3 兼容存储：

1. 保持 `STORAGE_TYPE=minio`
2. 把 `.env` 中 `MINIO_ENDPOINT / MINIO_ACCESS_KEY / MINIO_SECRET_KEY / MINIO_BUCKET` 改为外部存储配置
3. 在外部存储中预先创建 bucket
4. 确保对象有可访问策略，或后续自行改造成签名 URL

要注意：

1. 当前单机 Compose 仍会按默认方式启动本地 MinIO
2. 如果你希望“完全不启动本地 MinIO”，需要额外维护一份 Compose override，不属于当前仓库默认主线

### 5.7 数据库初始化与升级

第一次部署时：

1. 单机 Compose 的 MySQL 会在数据目录首次创建时自动导入 `sql/init.sql`
2. 如果你已经有旧数据卷，`init.sql` 不会重复执行

后续升级时：

1. 检查 `sql/migrations/` 目录
2. 按顺序执行新增 SQL
3. 当前仓库尚未接入 Flyway/Liquibase，升级流程需要人工确认

手工执行示例：

```bash
docker compose -f docker-compose.prod.yml exec mysql \
  mysql -uroot -p"${DB_PASSWORD}" qrcode_channel < sql/migrations/20260209_create_campaigns_table.sql
```

### 5.8 单机部署后的验收

验收时不要只看首页能不能打开，至少把下面这条链路走完：

1. 健康检查

```bash
curl http://127.0.0.1/api/health
```

2. 打开首页，确认前端正常加载
3. 在“配置管理”中创建并激活公众号/小程序配置
4. 创建活动
5. 生成二维码
6. 打开 `/jump?qid=<id>`，确认落地页可加载
7. 扫码后检查“扫码看板”是否新增记录
8. 打开二维码图片 URL，确认图片可访问

### 5.9 单机升级发布

升级步骤：

```bash
git pull
bash deploy.sh
```

如果只是重启服务：

```bash
docker compose -f docker-compose.prod.yml restart
```

### 5.10 单机回滚

单机环境的回滚比较直接，就是回退代码版本后重新部署：

1. 切回上一个 Git Tag 或提交
2. 重新执行 `bash deploy.sh`
3. 如果升级包含数据库变更，回滚前必须先确认数据兼容性

建议保留下面这几个习惯：

1. 每次上线前打 Git Tag
2. 每次上线前备份数据库
3. 数据结构变更单独审批

### 5.11 单机备份建议

MySQL 备份：

```bash
docker compose -f docker-compose.prod.yml exec mysql \
  mysqldump -uroot -p"${DB_PASSWORD}" qrcode_channel > backup-qrcode.sql
```

MinIO 数据备份：

1. 备份 Docker volume
2. 或定时把 bucket 数据同步到公司对象存储

### 5.12 单机常见问题

#### 页面能打开，但接口全失败

优先检查：

1. `frontend/nginx.conf` 是否已包含 `/api` 代理
2. `backend` 容器是否正常启动
3. `curl http://127.0.0.1/api/health` 是否通

#### 生成二维码成功，但图片打不开

优先检查：

1. `MINIO_ENDPOINT` 是否正确
2. bucket 是否存在
3. bucket 是否允许访问

#### 扫码跳转地址还是 localhost

优先检查：

1. `.env` 中 `APP_BASE_URL`
2. 是否重新部署了后端

#### 微信报 `invalid ip`

优先检查：

1. 公众号后台是否放行了后端出口 IP
2. 服务器是否经过了 NAT 或出口代理

## 6. K8S 集群部署（正式生产主线）

### 6.1 当前仓库已经补好的 K8S 物料

目录：`deploy/k8s/base/`

包含：

1. `namespace.yaml`
2. `configmap.yaml`
3. `secret.example.yaml`
4. `backend-deployment.yaml`
5. `backend-service.yaml`
6. `frontend-deployment.yaml`
7. `frontend-service.yaml`
8. `ingress.yaml`
9. `hpa.yaml`

### 6.2 K8S 前置条件

开始前先确认这些外部条件已经具备：

1. 已有可用 K8S 集群
2. 已有镜像仓库可推送镜像
3. 已有外部 MySQL
4. 已有外部 MinIO/OBS/S3 兼容对象存储
5. 已有域名和 Ingress Controller
6. 已有 TLS 证书或证书管理方式

### 6.3 K8S 配置原则

这套 K8S 基线约定如下：

1. 命名空间固定为 `qrcode`
2. 服务名固定为 `backend`、`frontend`
3. Ingress 路由：
   - `/api` -> backend:8080
   - `/jump` -> frontend:80
   - `/` -> frontend:80
4. 后端配置从 `ConfigMap + Secret` 注入
5. 对象存储主线写法为外部 OBS / S3 兼容

### 6.4 K8S 部署步骤

1. 创建命名空间

```bash
kubectl apply -f deploy/k8s/base/namespace.yaml
```

2. 修改并应用配置

需要先把下面两份文件按实际环境改好：

1. `deploy/k8s/base/configmap.yaml`
2. `deploy/k8s/base/secret.example.yaml`

建议这样处理：

1. `secret.example.yaml` 改名为内部实际文件后再 apply
2. 不要把真实 Secret 回写到 Git 仓库

3. 应用配置

```bash
kubectl apply -f deploy/k8s/base/configmap.yaml
kubectl apply -f deploy/k8s/base/secret.example.yaml
```

4. 构建并推送镜像

```bash
export TAG=v1.0.0
export REGISTRY=registry.example.com

docker build -t ${REGISTRY}/qrcode-channel/backend:${TAG} backend
docker build --build-arg VITE_API_BASE_URL=/api -t ${REGISTRY}/qrcode-channel/frontend:${TAG} frontend
docker push ${REGISTRY}/qrcode-channel/backend:${TAG}
docker push ${REGISTRY}/qrcode-channel/frontend:${TAG}
```

5. 先把 `deploy/k8s/base/backend-deployment.yaml` 和 `deploy/k8s/base/frontend-deployment.yaml` 中的镜像地址改成实际仓库地址，或者先按占位镜像 apply 再用 `set image` 覆盖。

6. 应用基础资源

```bash
kubectl apply -f deploy/k8s/base/backend-deployment.yaml
kubectl apply -f deploy/k8s/base/backend-service.yaml
kubectl apply -f deploy/k8s/base/frontend-deployment.yaml
kubectl apply -f deploy/k8s/base/frontend-service.yaml
kubectl apply -f deploy/k8s/base/ingress.yaml
```

7. 如需发布本次新镜像，执行：

```bash
kubectl -n qrcode set image deploy/backend backend=${REGISTRY}/qrcode-channel/backend:${TAG}
kubectl -n qrcode set image deploy/frontend frontend=${REGISTRY}/qrcode-channel/frontend:${TAG}
```

8. 如果集群已启用 Metrics Server，再按需开启 HPA

```bash
kubectl apply -f deploy/k8s/base/hpa.yaml
```

### 6.5 K8S 发布与观察

发布后先看滚动更新状态：

```bash
kubectl -n qrcode rollout status deploy/backend
kubectl -n qrcode rollout status deploy/frontend
```

查看 Pod：

```bash
kubectl -n qrcode get pods -o wide
```

查看日志：

```bash
kubectl -n qrcode logs deploy/backend --tail=100 -f
kubectl -n qrcode logs deploy/frontend --tail=100 -f
```

### 6.6 K8S 回滚

```bash
kubectl -n qrcode rollout undo deploy/backend
kubectl -n qrcode rollout undo deploy/frontend
```

回滚后再确认状态：

```bash
kubectl -n qrcode rollout status deploy/backend
kubectl -n qrcode rollout status deploy/frontend
```

回滚能否成立，取决于两个前提：

1. 上一版镜像必须仍在镜像仓库中可拉取
2. 如果数据库做了不兼容变更，应用回滚前要先评估数据兼容性

### 6.7 K8S 验收

K8S 环境建议按域名做验收：

1. `GET https://qrcode.company.com/api/health` 返回成功
2. 首页可打开
3. 配置管理可读写
4. 二维码生成成功
5. `/jump?qid=` 可打开
6. 扫码后看板有新增记录
7. 图片链接可访问

## 8. 发布前最终检查

正式上线前，最后再过一遍这份检查表：

1. `.env` 或 `ConfigMap/Secret` 是否与环境一致
2. `APP_BASE_URL` 是否为真实公网域名
3. `APP_CORS_ALLOWED_ORIGINS` 是否只放行需要的域名
4. MySQL 是否已初始化或执行迁移
5. 对象存储 bucket 是否存在
6. 微信配置是否已激活
7. 镜像 Tag 是否正确
8. 回滚路径是否明确
9. 数据库是否已备份

## 9. 推荐执行顺序

如果这是第一次把项目迁到公司环境，建议按这个顺序推进：

1. 本地验证前后端构建通过
2. 单机 Ubuntu 先跑通
3. 完成业务验收
4. 再迁入 K8S
5. 最后接入 Gitee CI/CD

这样做的好处很实际：

1. 先验证代码是否可运行
2. 再验证部署方式是否正确
3. 最后再验证自动化是否接通
