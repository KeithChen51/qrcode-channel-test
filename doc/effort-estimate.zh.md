
# 工程工作量预估（迁移到 Spring Boot 2 + Vue 3）

本文档对迁移 qrcode-channel-demo 到公司技术栈的工作量进行估算，
以人日（PD）计，给出 P50（最可能）
与 P90（保守）范围。

## 1. 假设与前提
- 功能范围保持与现有系统一致。
- 无重大 UI 重新设计，仅框架替换。
- 数据库仍为 MySQL。
- NAS 为测试环境，使用 Docker 部署。
- 认证未定：默认保留 OAuth，
  SSO 作为可选增量。
- 存储未定：默认适配现有逻辑，
  本地或 MinIO 作为可选增量。

## 3. 可选增量
- 对接公司 SSO：+5 ~ +10 人日
- MinIO/NAS 本地存储签名 URL：
  +3 ~ +6 人日
- UI 重新设计或新需求：+5 ~ +12 人日
- 新增报表/分析功能：+5 ~ +15 人日

## 4. 时间尺度（按周）
假设每人每周 5 人日。

1) 1 人全栈
- P50：约 14 周
- P90：约 23 周

2) 2 人（后端 + 前端）
- P50：约 8-10 周
- P90：约 12-16 周

3) 3 人（后端 + 前端 + QA/DevOps）
- P50：约 6-8 周
- P90：约 10-12 周

## 5. 关键路径
- 数据库迁移与 API 设计需优先
- 微信/二维码服务在关键路径
- 前端依赖稳定 REST API
- 端到端测试依赖回调与存储适配

## 6. 交付物
- Spring Boot 2 后端服务（REST API）
- Vue3 + Element Plus 前端
- Flyway/Liquibase 迁移脚本
- Docker 镜像与 NAS compose
- 核心流程测试报告

## 7. 风险
- 认证方案变更
- 存储方案影响二维码 URL
- 微信 API 限制或代理要求
- 回调接口兼容性

## 8. 进一步收敛估算的事项
- 明确认证方案（SSO 或 OAuth）
- 明确存储方案（NAS 本地或 MinIO）
- 确认是否保留未挂载页面
